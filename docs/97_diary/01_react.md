## 22.11.16 - 리액트에 관하여

여러 프로젝트를 리액트로 진행하면서 느낀 점들을 정리한 문서.

예전에는 기술의 트렌드, 회사의 채용 공고를 보면서\
해당 수준의 기술을 배우지 않으면 왠지 뒤쳐지는 것 같고,\
얼른 이런 것들을 배워야겠다라는 생각을 했고\
웹을 이렇게 작성하지 않으면 되게 올드하고, 나쁜 것처럼 편향된 사고로 생각했던 것 같다.

나는 회사에서 거의 2년 정도 Web UI 라이브러리를 개발 해왔다.\
기존 레거시 프로젝트가 순수한 js로 작성되어 있어 순수한 js 및 브라우저 DOM 조작에 맞는 라이브러리를 작성했다.

리액트로 일부 프로젝트도 완수 했으며,\
차세대 프로젝트를 리액트 + 일렉트론으로 진행할 구조를 설계하고 있다.\
회사에서는 웹 기술 리드 엔지니어 겸으로 웹으로 진행되는 모든 신규 프로젝트의 웹 개발 가이드를 하고 있다.

새 프로젝트에 리액트를 도입하였는데, 기존 개발자들은 대부분 리액트를 처음 접했다.\
처음에는 하나하나 헤딩하며 해나갈 수 있을거라는 막연한 생각으로 진행이 되었는데,
대부분 자바 기반 백엔드 엔지니어이고\
리액트는 구조를 **함수형 컴포넌트**로 가이드하게 되면서\
기존의 절차 지향, 개체 지향적인 관념에 익숙한 개발자가 프로그래밍을 하기에는\
어려운 점들이 많았다.

리액트에서는 상태의 흐름을 이해해야 했다.\
데이터의 단방향적인 흐름에서 변화를 hook하여 쓰는 구조인\
함수형 컴포넌트 구조와 hook API는 백엔드 개발자들에게는 친숙한 개념이 아니었다.

처음 접하는 개발자들은 리액트의 리렌더 동작 방식을 잘 생각하지 않았고\
특정 동작마다 UI의 속성들이 초기화 되는 리렌더 문제에 계속해서 질문했다.\
추가로 hook API에 대한 이해도가 낮아 쓰질 않으니 렌더링 성능은 나락으로 흘러가고 있었다.\
결국에는 리액트 부분을 전부 들어내고, 다른 환경으로 개발 방향을 바꾸기로 하였다.

## 무엇이 문제였는가?

막연히 헤딩하면서 리액트를 쓴다면 잘 따라올 것이라고 믿었던 부분이 문제였다.\
당연히 여러 자료를 보고, 기존의 사용된 패턴을 보면서 가이드를 하면 될 것이라 여겼지만,\
실상은 껍데기만 동작하는 프로토타입 수준의 UI만 구성되었다.

실제 데이터를 바인딩하고, 전역 상태 관리하는 부분마다\
신규 투입된 개발자들은 UI가 새로 리렌더되어 초기화되는 현상이나\
백엔드간 데이터 통신 처리에 난항을 겪었다.

나는 이를 위해서 내가 프로젝트의 프론트엔드 시니어로 투입되거나\
적어도 한 달의 교육 과정이 필요하다고 어필하였고 그 결과 리액트는 덜어내게 되었다.

게다가, 투입된 개발자들은 브라우저 엔진이나 HTML/CSS/JS에 대한 기본 이해도가 적고\
node의 환경도 친숙하지 않은 개발자들로 구성되어\
모든 것을 서포트해야 하는 입장에서 리액트까지 가이드하니\
관련 환경을 가이드하는 나로서는 더욱 더 지옥이었다.

그리고, 이 프로젝트는 하나의 WAS 인스턴스에서\
F/E 서비스와 B/E 서비스가 제공되어야 할 구조였다.\
B2B로 자바 기반의 환경을 가진 고객사를 주 타겟으로\
솔루션을 제공하는 입장에서 서비스 구조를 분리하여 제공하기에는 문제가 있는 프로트젝트였다.\
만약, 운영의 주체가 서비스 제공자였다면 문제가 없겠지만,\
운영의 주체가 고객사가 되므로 고객사 환경에서 설치가 편해야 한다는 조건이 있었다는 점.

이러한 구조에서 리액트로 웹 앱 환경을 제공한다면,\
WAS의 라우팅 설정을 조작하거나 멀티 페이지 앱을 리액트로 해야하는 기이한 구조가 될 수도 있었다.

정리하자면, 투입된 개발자들은 맨땅 헤딩으로 리액트를 하고 있었고,\
최종적인 제품의 모습에도 리액트를 억지로 끼워 맞춘 느낌이 든다는 것이다.

## 리액트를 쓰는 이유

작성해야될 App의 크기가 커지면 커질 수록 상태 관리의 난해함을 겪게될 것이고,\
재사용이 필요한 UI 컴포넌트들이 많이 필요하게 된다.

리액트는 이를 해결할 UI 기반을 제공해주며,\
node.js 기반 개발 환경에서 다양한 도구들과 결합하며 생산성과 안정성을 제공해준다.\
이를 통해 표준화된 프론트엔드 제품 개발 프로세스를 가져갈 수 있다.

다만, 단순히 리액트의 겉보기만을 보고 리액트 써야지는 금물로 보인다.\
내부적으로 node 기반 개발 환경 구성을 위해 이것저것 설정할 것이 많다.\
vite부터 linting, testing, styling 도구 결합과 빌드 파이프라인을 설계해야 한다.

CRA나 react-scripts등으로 쉽게 구성할 수도 있지만,\
디테일한 다양한 도구를 결합하려면 결국에는 혼자서 bundle 환경을 구축할 수 있어야한다.

단순히 리액트만을 껍데기만을 보고 사용하려는 생각은 좋지 않은 것 같다.\
CRA에 의존하거나, 누군가가 만들어준 개발 환경 구성에 의존적이라면\
나는 리액트 사용을 고민해보라고 말해주고 싶다.

## 언제 프론트엔드 프로젝트를 리액트로 적용하는게 좋을까?

아래에 해당되는 경우에는 긍정적인 검토로 해도 좋을 것 같다는 느낌이다.

- F/E 서비스와 B/E 서비스가 분리된 아키텍처
- F/E 개발자와 B/E 개발자가 분리된 프로젝트
- 다양한 프론트엔드 도구를 결합한 개발 환경에 친숙한 개발자 구성
- SPA를 요구하는 디테일한 기능적 조건 (UI와 웹 앱의 상태/흐름 제어)
- 애플리케이션의 복잡도 높으며, 높은 확장성이 요구되는 프로젝트

반대의 경우거나 이런 맥락을 이해하지 못 한다면,\
리액트 사용을 조금은 고민해봐야 하지 않을까.

## 결론

프론트엔드 프로젝트를 운영이나 구조적인 관점, 기존 개발 환경을 보지 않고,\
단순하게 기술적 트렌드로 방향을 잡고 처음에 가이드를 하게된 나의 경험을 토대로 작성한 내용이다.

다른 개발 경험을 가진 개발자들은 공감하지 못할 내용일 수도 있다.\
몸담고 있는 조직이 크거나 서비스가 거대하고 레거시한 개발 환경이 아니라면\
이러한 내용을 다르게 느낄 수도 있다고 생각한다.

리액트 같은 UI 라이브러리를 쓰지 않고서도\
복잡한 구조의 좋은 웹 앱은 충분히 만들 수 있다.\
생각해보면, 종종 큰 IT 서비스들은 UI 프레임워크를 쓰지 않고 개발된 모습을 볼 수 있다.\
내부 동작을 디테일하게 제어하고 완벽한 성능을 뽑아내려면 UI 프레임워크 사용은 오히려 오버헤드가 된다.

대부분의 신생 IT 서비스들이 UI 기반을 쓰는 이유는\
이러한 개발 환경이 업계 표준 기술처럼 자리를 잡아 이를 갖춘 경력자 공급이 빠르며\
그런 소양을 갖춘 개발자를 투입하여 빨리 개발을 진행할 수 있기 때문이라는 생각도 든다.

어느정도 정형화된 UI 기반 없이 디테일하게 하나하나 모두 설계하고\
개발하기에는 유지보수의 한계와 시간이 부족하니, 어쩔 수 없을 것이다.

이만 글을 마친다.
