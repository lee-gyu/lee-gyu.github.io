## 22.11.16 - 리액트에 관하여

여러 프로젝트를 리액트로 진행하면서 느낀 점들을 정리한 문서.

예전에는 기술의 트렌드, 회사의 채용 공고를 보면서\
해당 수준의 기술을 배우지 않으면 왠지 뒤쳐지는 것 같고,\
얼른 이런 것들을 배워야겠다라는 생각을 했고\
웹을 이렇게 작성하지 않으면 되게 올드하고, 나쁜 것처럼 편향된 사고로 생각했던 것 같다.

나는 회사에서 거의 2년 정도 UI 라이브러리를 개발 해왔다.\
React에 쓰이는 컴포넌트 라이브러리는 아니며, 기존 레거시 프로젝트가 순수한 js로\
작성되어 있어 순수한 DOM 조작에 맞는 라이브러리를 작성했다.

리액트로 일부 프로젝트도 완수 했으며,\
차세대 프로젝트를 리액트 + 일렉트론으로 진행할 구조를 설계하고 있다.\
회사에서는 기술 리드 엔지니어 겸으로 신규 프로젝트의 프론트엔드 부분을\
리액트로 가이드하며 신입/경력 대상으로 리액트 교육도 일부 진행했다.

새 프로젝트에 리액트를 도입하며 기존 개발자들은 대부분 리액트를 처음 접했는데\
여기서 문제가 시작했다. 대부분 자바 기반 백엔드 엔지니어가 대부분이고\
리액트는 **함수형 컴포넌트**가 주류로 자리 잡으면서\
기존의 절차 지향, 개체 지향적인 관념에 익숙한 개발자가 프로그래밍을 하기에는\
어려운 점들이 많았다.

리액트에서는 상태의 흐름을 이해해야 했으며\
이 상태 변화를 hook하여 쓰는 구조인 함수형 컴포넌트 구조는\
백엔드 개발자들에게는 친숙한 개념이 아니었다.

처음 접하는 개발자들은 리액트의 리렌더 동작 방식을 잘 생각하지 않았고\
특정 동작마다 UI의 속성들이 초기화 되는 리렌더 문제에 계속해서 질문했다.\
추가로 hook API에 대한 이해도가 낮아 쓰질 않으니 렌더링 성능은 나락으로 흘러가고 있었다.\
결국에는 리액트 부분을 전부 들어내고, 서버사이드 렌더링 방식으로 웹 개발로 방향을 바꾸기로 하였다.

## 무엇이 문제였는가?

기존 회사에서 신규 프로젝트로 리액트를 쓰고자 한다면 가장 중요한 것이 **_아키텍처_**라고 생각한다.\
F/E 서비스와 B/E 서비스가 분리되어 각 서비스에 대한 개발자 그룹을 분리하고\
프론트엔드 개발자 팀과 백엔드 개발자 팀으로 나누어 운영한다면 좋은 선택이라고 볼 수 있을 것 같다.

현재 리액트를 진행하고 있는 조직은 적합한 구조는 아니었다.\
신규로 도입하는 프로젝트에서 각 개발자의 구분 없이 모두가 프론트와 백엔드를 해야하는 상황이었다.\
해당 프로젝트를 진행할 개발자는 2~3명 남짓으로\
이 상황에서 리액트를 도입하니, 1명만 리액트 부분을 맡게 되니,\
추후 프로젝트 유지보수에 적신호를 보여주는 상황이 될 수 있었다.\
~~물론, 그 1명도 리액트를 제대로 사용하지 못하고 계속 질문만 하는 상황이었다...~~

게다가, 브라우저 엔진이나 HTML/CSS/JS에 대한 기본 이해도가 적고\
node의 환경도 친숙하지 않은 개발자들로 구성되어\
모든 것을 서포트해야 하는 입장에서 리액트까지 가이드하니\
관련 환경을 가이드하는 나로서는 더욱 더 지옥이었다. (매주 1~2회 약 2시간 가량의 코드 리뷰...)

그리고, 이 프로젝트는 아키텍처 상 하나의 WAS 인스턴스에서\
F/E 서비스와 B/E 서비스가 제공되어야 할 구조였다.\
B2B로 자바 기반의 환경을 가진 고객사를 주 타겟으로\
솔루션을 제공하는 입장에서 서비스 구조를 분리하여 제공하기에는 문제가 있는 프로트젝트였다.\
만약, 운영의 주체가 서비스 제공자였다면 문제가 없겠지만,\
운영의 주체가 고객사가 되므로 고객사 환경에서 설치가 편해야 한다는 조건이 있었다는 점.

이러한 구조에서 리액트로 웹 앱 환경을 제공한다면,\
WAS의 라우팅 설정을 조작하거나 리액트 내부 라우팅 기능을 임의로 구성해야하는\
기이한 구조가 될 수도 있다.\
운영 환경 Web Server의 설정과 라우팅 처리에서 충돌이라도 난다면 더욱이 더 골치아픈 상황이 될 수도 있고 말이다.

## 리액트를 쓰는 이유

node.js 개발 환경에서 주는 편리한 도구와의 결합이 가장 큰 이유로 보인다.\
모던한 프론트엔드 개발 환경을 완벽히 이용할 수 있으며\
다양한 도구들을 통해 개발 생산성을 끌어올려줄 수 있다.\
아래의 항목들이 쓰는 이유로 볼 수 있는 내용으로 생각된다.

- 상태와 UI 분리 구조의 Application 개발 (*상태 변화에 따른 단방향 UI 렌더 구조 적용)
- Single Page Application 작성
- 애플리케이션의 구조가 큰 경우 (컴포넌트 재사용성, 많은 모듈 관리 등을 고려할 큰 구조)
- Typescript를 결합한 프론트엔드 개발 환경
- Bundler를 통한 코드 트랜스파일링 (브라우저 호환성 처리 및 클라이언트 코드 난독화/패키징)
- 생산성 높은 스타일링 도구 결합 (tailwindcss, postcss, sass)
- 배포 버전의 소스 코드 압축 및 난독화
- 자동화 테스팅 도구와의 결합
- 다양한 리액트 라이브러리로 생산성 향상
  - swr
  - jotai
  - framer-motion
  - styled-component

단순히 리액트만을 보고 적용한다는 생각은 좋지 않은 것 같다.\
CRA에 의존하거나, 누군가가 만들어준 개발 환경 구성에 의존적이라면\
나는 리액트 사용을 고민해보라고 말해주고 싶다.

## 언제 프론트엔드 프로젝트를 리액트로 적용하는게 좋을까?

아래에 해당되는 경우에는 긍정적인 검토로 해도 좋을 것 같다는 느낌이다.

- F/E 서비스와 B/E 서비스가 분리된 아키텍처
- F/E 개발자와 B/E 개발자가 분리된 운영 조직
- Single Page Application으로 웹 앱의 상태가 유지되어야 하는 기능적 조건
- 애플리케이션의 복잡도 높으며, 높은 확장성이 요구되는 프로젝트
- 다양한 프론트엔드 도구를 결합한 개발 환경 활용
- 기본적인 웹 기술 지식이 습득된 상태 (브라우저 엔진 이해, 웹 보안 등)

반대의 경우거나 이런 맥락을 이해하지 못 한다면,\
리액트 사용을 조금은 생각해봐도 좋지 않을까.

## 결론

프론트엔드 프로젝트를 운영이나 구조적인 관점, 기존 개발 환경을 보지 않고,\
단순하게 기술적 트렌드로 방향을 잡고 결정하게된 나의 경험을 토대로 작성한 내용이다.

다른 개발 경험을 가진 개발자들은 공감하지 못할 내용일 수도 있다.\
몸담고 있는 조직이 크거나 서비스가 거대하고 레거시한 개발 환경이 아니라면\
이러한 내용을 다르게 느낄 수도 있다고 생각한다.

리액트 같은 UI 프레임워크를 쓰지 않고서도\
큰 구조의 좋은 퀄리티의 웹 앱은 충분히 만들 수 있다.\
생각해보면, 종종 큰 IT 서비스들은 UI 프레임워크를 쓰지 않고 개발된 모습을 볼 수 있다.\
내부 동작을 디테일하게 제어하고 완벽한 성능을 뽑아내려면 UI 프레임워크 사용은 오히려 오버헤드가 될 것이다.

대부분의 신생 IT 서비스들이 UI 프레임워크를 쓰는 이유는\
좋은 개발 도구의 지원 없이 디테일하게 하나하나 모두 설계하고\
개발하기에는 유지보수의 한계와 시간이 부족하니,\
UI 프레임워크와 여러 도구를 결합하여 활용하는 것 뿐이라는 생각이다.

이만 글을 마친다.
