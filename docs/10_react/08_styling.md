## Styling

React 컴포넌트에 스타일 서식을 입히는 여러 방식을 소개.\
큰 분류로 정적으로 css 스타일을 생성하는 방식과\
javascript에서 동적으로 필요한 스타일을 계산하여 로드하는 방식이 있다.

확장성이나 성능에 따라 어떤 것을 선택할지 달라진다.\
모든 문제를 해결해주는 만능 도구는 없으며, 결국 개발자의 선택이다.

## sass (scss)

sass의 명세를 구현한 빌드 도구를 `LibSass`와 `Dart Sass` 2가지가 있다.\
`LibSass`의 경우 유지보수가 중단되어 `Dart Sass`사용을 권하고 있다.\
성능면에서는 `LibSass`가 더 빠르지만, 문법이나 C/C++로 작성되어 유지보수의 한계를 맞은 듯하다.

그러므로, 풍부한 문법을 위해서는 `Dark Sass`를 이용하자.

sass 파일과 scss파일이 존재하는데, sass의 경우 `{ }`와 같은 블록 없이 사용하고,\
scss는 css처럼 블록을 같이 사용한다.\
내 주관적인 관점에서는 css에서 확장된 느낌을 주는 scss를 더 선호한다. (css를 그대로 옮겨도 쓸 수 있으니 말이다.)

- 장점
  - 범용성이 좋다. (React든, Vue든, 일반 웹이든, css만 link 또는 import하면 된다)
  - 학습 곡선이 낮다.
  - post-css 같은 css 후처리 도구와 연계가 가능하다.
  - 반복적인 css 스크립팅을 줄이는 다양한 문법 제공 (mixin, include, for, nesting, modularity, extend, operator)
- 단점
  - scss를 처리하는 빌드 성능이 빠르지 못하다.
  - 개발 구성 환경에서 복잡도가 증가한다.
  - 파일이 많고, 스타일의 구조가 복잡해질 수록, 유지보수가 어려워진다.

## styled-components

javascript 코드를 통해 CSS 스타일을 정의하는 방식.\
코드 상에서의 상속, 합성 등을 이용하여 다양하게 스타일을 응용하여 컴포넌트를 작성할 수 있다.\
번들 도구와의 복잡한 설정은 필요 없으며, 페이지 로딩 시 필요한 스타일을 계산하므로,
페이지 성능에 영향을 주는 큰 단점이 있다. (그런 점에서 나는 이것을 스타일링 도구로 선호하지 않는다.)

페이지 성능에 디테일한 부분까지 신경을 쓰는 개발자라면 쓰는 것이 꺼려질 수 있다.

- 장점
  - 리액트 네이티브와도 스타일 호환이 가능하다.
  - 재사용성과 확장성이 높다.
  - 동적으로 다양한 스타일링을 연출할 수 있다.
  - 번들 도구와의 복잡한 설정이 필요 없음.
- 단점
  - 런타임 성능에 영향을 미친다. (리렌더, 페이지 초기화 등)  
    (안 그래도 무거운 리액트의 무게를 가중한다. 가장 치명적인 단점이다.)
  - 번들되는 js 크기 증가

## post-css

post-css는 css 결과에서 여러 도구들과 연계하여 브라우저 호환성이나,\
다양한 커스텀 스타일을 사용하게 하는 후처리 도구로 사용한다.

그 예로 들면 아래와 같은 기능들을 사용할 수 있다.

- svg를 inline svg로 변환. (이 경우, css 하나에 svg 파일들을 패키징하는 이점)
- 브라우저 접두사 별 분기 클래스 생성 (-webkit, -moz 등)
- 클래스 이름 local scope화 (글로벌 클래스 이름 충돌을 회피)
