## 문자 인코딩이란?

컴퓨터의 모든 데이터는 0과 1로 구성된다.\
문자로 작성하는 이러한 글도 결국에는 0과 1로 구성되어야 한다.\
특정 데이터를 표현하는 일종의 약속을 **인코딩 표준**이라고 부른다.\
문자 체계에서는 대표적으로 ASCII와 Unicode가 있다.

아스키는 7비트 데이터에 대한 인코딩으로 알파벳과 특수 문자 등을 표현한다.\
컴퓨터가 개발된 초기에는 각 문자권마다 고유의 표준을 사용했다.\
한글은 CP-949, EUC-KR 등을 사용하여 이는 국제 소프트웨어를 개발하는 회사들에게 문제가 되었다.

이러한 어려움을 해결하고자 유니코드라는 새로운 표준이 만들어지며, "Uni"라는 접두사가 나타내듯\
모든 언어의 문자를 하나의 표준에 담겠다는 목표로 제정되었다.\
유니코드에서 한 문자는 최대 32bit로 표현된다.\
즉, 하나의 문자에 42억개를 나타낼 수 있다.\
이를 이용해 한글, 한자, 히라가나 등에 요즘에는 이모지도 유니코드에 포함되고 있다.

## ASCII

초창기 영어권 문화에서 7bit 체계로 표현하는 표준.\
추후 전세계에서 컴퓨터를 사용하면서 각 나라의 문화권 문자를 표현하는 체계에 문제가 생김.\
한자, 한글, 일본어 표기에 문제 7bit 체계의 한계가 발생.

## 한글 인코딩

한글 자판 입력 방식 중, 조합형과 완성형 중 어떤 것이 더 나은가를 두고 초기 논쟁이 있었다고 함.\
한글 창제 방식 상, 조합형이 더 부합하다고 하였으나 완성형으로 채택됨\
한글의 경우 초성, 중성, 종성을 통해 가능한 경우의 수가 매우 많이 나옴.\
이는 한자를 표시해야하는 중국, 일본에서도 큰 문제가 되었음. (한자에 필요한 모든 문자는 최소 10만개가 넘는다고 함.)

- 조합형: ㄱㅏㅁ (1 + 2 + 4) 방식으로 '감'이라는 글자를 만들자.
  - 비트를 많이 필요로 하는 단점과 다른 문자 체계와 호환이 문제가 있어 결국 채택되지 않음
- 완성형: 501 코드는 '감'이라는 글자로 이용하자
  - CP-949: 사용빈도가 높은 2350자만을 채택하여 지원
  - EUC-KR: 2350자에서 지원하지 못하는 8822자를 추가한 방식. (CP-949가 사용하던 영역을 침범하여 호환이 제대로 안됨)
  - 윈도우즈는 CP-949가 기본 표준으로 쓰이고 있다.

## 어디서 문제가 되었는가?

Web에서는 다양한 OS의 다양한 브라우저에서 실행되는 환경이다.\
한글 고유의 문자 체계인 CP-949, EUC-KR은 이때 문제가 되므로, 웹에서는 UTF-8 체계로 통합하게 된다.

## 유니코드

모든 국가 언어 문자를 표현하기 위해 나온 표준.\
현재 유니코드에서는 완성형 11,172자와 조합형 한글 낱자가 모두 수록되어 표현할 수 있다고 한다.
(한글은 U+AC00 ~ D7A3 영역으로 배치)

유니코드는 문자 하나를 4바이트 테이블에 배당하므로, 이걸 그대로 사용하게 되면\
텍스트 파일 용량이 어마어마하게 커지는 문제가 있다.\
이걸 보완하기 위해 자주 쓰는 문자 테이블을 1바이트(UTF-8) 또는 2바이트(UTF-16)로 표현하는 대신,
자주 쓰이지 않는 문자는 더 많은 바이트로 처리하게 된다.

이러한 유니코드 바이트로 표기하는 방식인 UCF, UTF-8, UTF-16 등의 여러 문자 처리 방식이 있다.

## UTF-8

ASCII에 해당되는 문자를 사용하는 경우 1바이트로 처리하고,\
unicode 체계에서 2byte가 필요한 문자는 3바이트로 처리한다.\
UTF-16이나 EUC-KR, CP-949는 한글을 2바이트로 처리하지만, UTF-8은 3바이트로 처리하여\
UTF-8의 경우 용량이 더 필요한 단점이 있긴하다.\
다만 일반 아스키 코드 문자를 1byte로 처리하는 장점이 있어, 웬만한 경우는 UTF-8을 이용하는 것이 표준이 되었다.

```text
가나다라
EA B0 80 EB 82 98 EB 8B A4 EB 9D 8C
1234
31 32 33 34
abcd
61 62 63 64
```

UTF-8에서 가(EA B0 80) 나 (EB 82 98) 다 (EB 8B A4) 라 (EB 9D 8C)로 처리된 결과를 볼 수 있다.

### UTF-8 규칙

- 첫 시작 비트가 0을 할당하면, 나머지 7비트를 기존 아스키 코드를 할당하여 표현 (0xxx xxxx)
- 사용하는 바이트가 2바이트가 넘을 때, 2바이트 표식 데이터의 충돌을 고려하여 10 비트를 넣어준다.

| 자릿수  | 코드값 범위        | 첫 바이트 | 둘째      | 셋째      | 넷째      |
| ------- | ------------------ | --------- | --------- | --------- | --------- |
| ~7비트  | 0 ~ 0x7F           | 0xxx xxxx |           |           |           |
| ~11비트 | 0x80 ~ 0x7FF       | 110x xxxx | 10xx xxxx |           |           |
| ~16비트 | 0x800 ~ 0xFFFF     | 1110 xxxx | 10xx xxxx | 10xx xxxx |           |
| ~21비트 | 0x10000 ~ 0x1FFFFF | 1111 0xxx | 10xx xxxx | 10xx xxxx | 10xx xxxx |

이론상 31비트까지 되어 있지만, 현재 0x10FFFF 범위까지만 사용한다고 한다.

### UTF BOM (Byte Order Mark)

EF BB BF라는 UTF-8이라는 문서라는 것을 보여주는 데이터가 먼저 헤더로 보여지는 문서\
이것을 설정한 문서를 스크립트로 처리하면 문제가 될 수 있으니, 스크립트 파일은 BOM을 지정하지 않는 것이 좋다고 한다.

## UTF-16

각 문자 단위를 2바이트 체계로 처리하기 때문에 Little Endian, Big Endian인지를 표기를 해야하는 문제가 생긴다.\
게다가 영문도 2바이트로 처리해야하므로 아스키 코드에 해당하는 문자가 많을 수록 바이트 손해가 생긴다.\
한글이나 한자의 경우 2바이트로 처리하여 UTF-8에 비하면 바이트 손해가 생기지 않는다.

BOM으로 가장 헤더의 데이터가 FF FE라면 Little Endian\
FE FF라면 Big Endian으로 처리된다.

Java/닷넷은 기본으로 UTF-16을 사용하며, 모두 Big Endian으로 처리한다.\
그래서 이모지를 문자로 이용한다면 char[2]로 써야한다.

UTF-16은 바이트 순서를 어떻게 처리하는지를 꼭 읽어야하는 문제가 있다.

- Little Endian: 경우 0x12345678을 78 56 34 12와 같이 각 바이트 주소를 내림차순으로 처리하는 방식
- Big Endian: 0x12345678을 12 34 56 78로 각 바이트 주소를 오름차순으로 처리하는 방식
