## Process & Thread

OS가 하는 리소스 관리 단위인 Process / Thread

## Process

최소 1개의 쓰레드를 가지는 프로그램 처리 단위.

코드의 연산을 처리하기 위해 가장 먼저 필요한 리소스는 CPU\
연산 처리 후 결과를 저장하기 위해 그 다음으로 필요한 리소스는 메모리

연산 결과는 바로 물리적인 메모리 공간으로 처리하지 않고 추상화된 저장 공간인 가상 메모리로 처리된다.\
프로세스는 필요한 가상 메모리를 할당 받으며, 가상 메모리는 1차 메모리(RAM)와 2차 메모리(HDD)로 추상화되어\
컴퓨터의 자원 현황에 따라 저장된다.

OS는 프로세스를 생성할 때 가상 메모리 공간을 할당함.\
프로세스가 죽거나 종료되면 해당 가상 메모리는 회수된다.\
프로세스 내의 Thread는 가상 메모리 안의 메모리만 접근되도록 제한된다.

## Thread

하나의 코드의 실행 흐름을 처리하는 단위\
한 프로세스 내에서 여러 쓰레드를 처리하는 것을 multi-threading이라고 함.\
(여러 프로세스 처리를 multi-tasking)

각 쓰레드는 각자의 stack 메모리를 저장하기 위해 TLS(Thread Local Storage)를 할당 받는다.

멀티 쓰레딩을 처리를 위해 고려해야되는 것은 2가지

- 동시성: 각 프로세스, 각 쓰레드의 연산이 동시에 일어나는 성질
- 동기화: 여러 쓰레드가 동시에 같은 메모리 주소의 Read/Write가 일어날 때, 이를 오류 없이 처리하는 방법
  - Mutex(i.e. Mutual Exclusion) Lock
    - 공유된 자원을 여러 쓰레드가 접근하는 것을 막음
    - 먼저 선점한 쓰레드가 임계 영역에 들어갈 때 해당 블럭을 lock 빠져 나올 때 unlock
  - 세마포어
    - 동시에 접근 가능한 쓰레드가 1개 이상일 때 사용
    - wait: 임계 영역에 접근 가능한 상태인지 확인
    - signal: 임계 영역에서 빠져나올 때 notify

## 멀티 프로세싱 vs 멀티 쓰레딩

크롬 브라우저는 멀티 프로세싱 모델로 각 탭마다 프로세스를 가진다.\
멀티 프로세싱과 멀티 쓰레딩은 어떤 때에 어느 방법이 효율적이고 선택해야하는가?

- 멀티 프로세싱
  - 각 작업 단위마다 서로 공유해야될 메모리 정보가 적은 경우 적합
  - 메모리 공유를 위해 별도의 외부 인터페이스 방식을 구현해야 한다. (네트워크 또는 IPC 등)
  - 프로세스 내부 처리 데이터를 캡슐화
  - 도커는 각 컨테이너를 프로세스 단위로 관리 한다.
- 멀티 쓰레딩
  - 프로세스 내 쓰레드끼리 메모리가 공유될 필요가 있는 경우
  - 이 경우, 동시성에서 각 메모리를 처리할 때 동기화에 처리를 고려해야한다.
